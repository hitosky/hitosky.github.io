浏览器解析Html,Js,Url

HTML解析

首先明白三个概念：

  1. 字符实体： 字符实体是一个转义序列，它定义了无法在文本内容中输入的单个字符或符号，一般以&开头，后接预定义的实体名称，或是#与十进制数。如’<‘的字符实体名称为’&lt’，实体编号为’&#60’。
  2. HTML字符实体 就像程序语言的关键字一样，HTML字符实体是浏览器为了解析html状态而预留的保留符号，常见的有’<‘与’>‘。
  3. 字符引用 字符引用又分为‘字符值引用’和‘字符实体引用’,字符引用是为了告诉浏览器，字符实体的转义序列有多长，一般用’;’来标志结束，如’<‘的字符值引用为’&#60 ;’，字符实体引用为’&#lt ;’

HTML解析器作为一个状态机，从输入流中获取字符并按照转换规则转换到另一种状态。解析过程中任何时候，只要遇到’<’符号（后面不跟’/’）就会进入’tag open state’,然后转到’tag name state’,然后转到’before attribute name state’….最后进入’data state’,并释放当前标签的token。 以<div>为例：

输入流 解析状态(从上至下)
开始 tag close state
< tag open state
d tag name state
i before attribute name state
v before attribute name state
> data state,release token
</div tag close state

‘<’与‘>’是浏览器解析html状态的重要标志，或者说解析状态的迁移条件是遇到’<‘或’>’ , ‘&lt’或者’&gt’虽然会被解码成’<’或’>’,却不会被浏览器当成是状态转移的条件。它们只会被当成用户输入的‘数据’

浏览器对于‘字符引用’的解析又分为三种情况:

  1. data state中的字符引用
  2. RCData state中的字符引用
  3. 属性值状态中的字符引用

情况1：

<div>这里的字符引用属于data state</div>

情况2： 这其实也属于1中的特殊情况，只是标签名为textarea、title、code、pre等为了让用户输入文本内容的标签。在这种状态的如果浏览器遇到’<’，并不会进入tap open state，而是进入了‘RCData小于号状态’，如果’<’后面不跟’/’符号，那么会重新进入’RCData state’。因此在这些标签中创建<script>标签，脚本不会执行。

情况3： 标签中内嵌的属性值

总结下：html解析状态根据’<’与’</>’来改变，要让<script>中脚本执行，前提条件是能使解析状态达到data state

URL解析

URL解析也是一个状态机模型，从输入流中进来的字符可以引导URL解析器转换到不同状态。 但是URL解析的字符必须是ASCII字母，不然会进入“无类型”状态，也就是说如果对URL协议中字符进行任何编码操作，解析器就无法对它进行正确的解码。

<a href="%6a%61%76%61%73%63%72%69%70%74:%61%6c%65%72%74%28%31%29">click me</a>

这段代码本义是为了执行动作：javascript:alert(1) ,但是由于进行了unicode编码，解析器无法对其进行解码。 同理

<a href="javascript%3aalert(1)>click me</a>

这段代码也无法执行对应的js动作，因为’%3a’无法被url解析器解析成’:’。 再看这段代码：

<a href="&#x6a;&#x61;&#x76;&#x61;&#x73;&#x63;&#x72;&#x69;&#x70;&#x74;:%61%6c%65%72%74%28%32%29">click me</a>

这段代码能正确执行相应的js动作，为什么？ 对于整个html文档来说，浏览器解析时候通常先用html解析器去解析，所以上面的代码会现在浏览器html解析器中解析，然后交由url解析器。经过html解析器解析后的相应代码就变为

<a href="javascript:%61%6c%65%72%74%28%32%29">click me</a>

当url解析器看见javascript:的时候，会把代码交由javascript解析器去解析，所以上面的代码最终被解析成

<a href="javascript:alert(1)">click me</a>

javascript解析

javascript解析器不同于html解析器，<script&gt标签有个特殊的属性：在它内部的字符引用不会被解析和解码。 下面的代码不会被执行

<script>&#97;&#108;&#101;&#114;&#116&#40;&#57;&#41;&#59</script>

javascript解析器会对类似”\uXXXX”unicode编码的字符进行解析，但是也分情况。

  1. 字符串中出现”\uXXXX”
  2. 标识符中出现”\uXXXX”
  3. 控制字符中出现”\uXXXX”

情况1： 当Unicode转义序列存在于字符串中时，它只会被解释为正规字符，而不是单引号，双引号或者换行符这些能够打破字符串上下文的字符。这项内容清楚地写在ECMAScript中。因此，Unicode转义序列将永远不会破环字符串上下文，因为它们只能被解释成字符串常量。 如下代码的”\u000a”会被解析成”\n”

<script>alert('1\u000a')</script>

情况2： 当Unicode转义序列出现在标识符名称中时，它会被解码并解释为标识符名称的一部分,例如函数名，属性名等。

<script>\u0061\u006c\u0065\u0072\u0074(1);</script>

上面的这段代码能被js解析器解析成

<script>alert(1);</script>

并且成功执行。

情况3： 这里说的控制字符是指单引号，双引号，圆括号，花括号等用来控制程序进行的字符，类似于html字符实体与一般字符实体的区别，”\uXXXX”无法取代相应控制字符的角色作用。

js解析器对于上面三种情况，总结下就是： Unicode转义序列只有在标识符名称里不被当作字符串，也只有在标识符名称里的编码字符能够被正常的解析。
