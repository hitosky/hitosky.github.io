JS变量对象与执行上下文

执行上下文(execution context)

JS在确定作用域链(scope link)与原型链(prototype link)的过程中需要明确当前环境的执行上下文。
JS在执行期间会产生一个栈，用以存放代码运行过程中产生的执行上下文。其结构类似于：
ECStack=[];
在每次进入一个函数(包括递归调用、构造函数与eval函数)的时候，会把相应的执行上下文压入ECStack，在代码运行完毕后会执行上下文会被弹出栈，其第一个元素就是当前活跃的上下文activeContext。

全局代码

在初始化阶段（程序启动），ECStack看起来像这样

ECStack=[  
 globalContext//全局上下文   
];

全局代码不包含在任何函数中。

函数代码

在进入函数代码的时候，会产生新的执行上下文并将其压入ECStack中。
比如下面的例子：

function foo(){
  alert('foo');
}
foo();

//当运行到foo()的时候
ECStack=[
  <foo>functionContext,
  globalContext
]
//当foo()执行完后
ECStack=[
  globalContext
]

functionContext弹出一般有两种可能：
1.函数返回
2.抛出异常却没有被捕获

eval代码

eval代码在被调用的过程中，不仅会产生evalContext，还会产生调用它的callingContext。

eval('var a=10');
/*
ECStack=[
  {context:evalContext,
  callingContext:globalContext},
  globalContext
]
*/

(function foo(){
  eval('var y=10');
})()
/*ECStack=[
  {context:evalContext,
  callingContext:<foo>functionContext},
  <foo>functionContext
]*/

变量对象(variable object)

JS会根据当前执行上下文，生成一个属性，叫做变量对象(variable object,简称VO),它存储了

  • 变量（var,变量声明）
  • 函数声明（function declaration,简称FD）
  • 函数形参

我们用VO={}表示变量对象，VO是执行上下文的属性

activeExecutionContext={
  VO:{
  },
}

例子：

var a=10;

function test(x){
  var b=20;
}
test(30);

对应的变量对象是：

VO(globalContext)={
  a:10;
  test:<reference to function>
};

VO(<test>functionContext)={
  x:30,
  b:20
};

变量对象的确定与修改主要分两个阶段：
1.进入执行上下文（代码执行之前）
2.代码执行阶段

刚进入执行上下文阶段

进入执行上下文时，变量对象的属性的确定按优先级（高优先级属性会覆盖低优先级的同名属性）从高到低分为

  1. 函数形参（如果是函数上下文）：
    对于有值传递的形参，VO会产生形如形参名:形参值的属性；对于没有值传递的形参，VO会产生形如形参名:undefined的属性
  2. 函数声明（FD）
    VO会产生函数名:函数引用的属性,注意不是函数表达式(FE)
  3. 变量声明（var)
    VO会产生变量名:undefined的属性,注意：必须是var声明的变量
    例子：
function test(a,b){
  var c=10;
  function d(){}
  var e=function _e(){};
  (function x(){});   // 函数表达式FE所以不会影响VO
}
test(10);

在以a=10传递test时：

VO(test)={
  a:10,
  b:undefined,
  c:undefined,
  d:reference to FD 'd',
  e:undefined
};

代码执行阶段

还是以上面的例子为例：

var c=10;
//代码运行到此处后，VO['c']=10
var e=function _e(){};
//代码运行到此处后，VO['e']=<reference FD '_e'>

关于‘覆盖’问题，让我们再来看个例子：

alert(x);  //function
var x=10;
alert(x);  //10
x=20;
function x(){}
alert(x);  //20

第一个alert(x)为什么是function？
首先程序刚运行到此上下文的时候，由于函数声明会覆盖同名变量声明，所以生成了VO={x:<reference to function 'x'>},因此当运行到第一个alert(x)时候显示function。
第二个alert(x)与第三个alert(x)，分别输出10与20是因为在‘代码运行阶段’VO['x']被分别修改为10与20。
又一个例子：

if(true){
  var x=10;
}
else{
  var y=10;
}
alert(x);   //10
alert(y);   //undefined,not because y is not defined

最后一个例子：

alert(x);
eval('var x=10');
alert(x);

第一个alert(x)会出错，显示ReferenceError: x is not defined,因为eval的有它自身的执行上下文，当前的VO不会去eval的上下文中寻找x，注释掉第一个alert(x)后，在eval之后生成了x，所以能正确弹框显示10。

弄懂VO对于确定作用域链、原型链以及明确this的指向非常有帮助。
